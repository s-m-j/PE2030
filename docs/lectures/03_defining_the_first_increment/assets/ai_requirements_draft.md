# Предложение AI: документ требований для пилотного этапа

> **Роль в лекции:** показывается в блоке 1 сценария ([../script.md](../script.md)) как готовый результат — без промпта и без комментариев. Разбор того, для кого документ фактически написан, открывается в блоке 2.

## Промпт

```
Опиши требования и критерии приёмки для пилотного этапа проверки данных
экологического мониторинга: нужно проверить качество данных существующих
датчиков и сопоставить их с историей жалоб жителей за последний месяц.
```

## Ответ AI (как показывается студентам)

### Пользовательские истории

**US-1.** Как специалист экологической службы, я хочу видеть долю пропущенных измерений по каждому датчику за последние 30 дней, чтобы оценить полноту данных перед анализом.

**US-2.** Как специалист экологической службы, я хочу видеть периоды, где показания датчика статистически расходятся с ближайшей официальной станцией, чтобы выявить потенциально неисправные или некалиброванные датчики.

**US-3.** Как специалист экологической службы, я хочу получить сводную таблицу по всем датчикам с индикацией «надёжный / требует проверки / недостаточно данных», чтобы приоритизировать дальнейшую работу с данными.

**US-4.** Как специалист экологической службы, я хочу сопоставить временные метки жалоб жителей с показаниями ближайшего датчика в соответствующий период, чтобы понять, коррелируют ли жалобы с измеримыми отклонениями.

### Критерии приёмки

- Для каждого датчика рассчитана доля пропущенных измерений за 30 дней.
- Реализован статистический тест расхождения показаний датчика и ближайшей официальной станции (например, сравнение распределений).
- Сформирована сводная таблица статусов датчиков.
- Для каждой жалобы за последний месяц найден ближайший по времени и расстоянию датчик и указано его показание.
- Результаты сохранены в структурированном виде (CSV/таблица), пригодном для дальнейшего анализа.

### Данные

| Поле | Источник | Комментарий |
|---|---|---|
| `station_id` | датчики, официальные станции | идентификатор источника |
| `pollutant`, `value`, `unit` | датчики, официальные станции | измеренное значение |
| `measured_at` | датчики, официальные станции | время измерения |
| `complaint_id`, `complaint_time`, `complaint_location` | реестр жалоб | из журнала обращений жителей |

### Технические заметки

Рекомендуется реализовать как аналитический скрипт (Python, pandas) без отдельного пользовательского интерфейса — результат представляется в виде таблиц и графиков для внутреннего анализа.

## Что скрыто за этим документом (не показывать до блока 2 — материал для разбора)

Документ структурирован, полон конкретики и на первый взгляд готов к работе. Проблема не в качестве формулировок, а в том, для кого он фактически написан.

- **Все четыре пользовательские истории написаны от лица одной аудитории** — специалиста экологической службы. Промпт не указывал аудиторию явно, но AI молча выбрал её и последовательно провёл этот выбор через весь документ.
- **Руководитель центра нигде не упомянут**, хотя именно ему через шесть недель нужно показать результат горсовету. Документ не содержит ни одного артефакта, который можно предъявить нетехнической аудитории.
- **Жители и общественный совет не упомянуты** ни разу, хотя исходная проблема — их жалобы. Пользовательская история US-4 обрабатывает жалобы как входные данные для внутреннего анализа специалиста, а не как то, на что нужно дать понятный ответ автору жалобы.
- **ИТ-отдел не упомянут**, хотя «структурированный вывод CSV» и «аналитический скрипт» — уже implicit-архитектурное решение, которое стоило бы явно согласовать с их требованием не создавать новых зависимостей.
- **Технические заметки незаметно превращаются в архитектурное решение** («реализовать как аналитический скрипт без интерфейса») — документ, который должен был описывать требования, по ходу дела уже выбрал форму реализации, не называя это выбором.

Ни один из этих пропусков не виден при первом чтении — документ выглядит профессионально и последовательно именно потому, что последовательно проведён с точки зрения одной, молча выбранной аудитории.

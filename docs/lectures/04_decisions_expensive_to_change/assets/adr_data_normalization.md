# ADR (пример): нормализация показаний разнородных источников данных

> **Роль в лекции:** контрпример к [`ai_generated_pilot_code.md`](ai_generated_pilot_code.md), используется в блоках 4 и 6 сценария ([../script.md](../script.md)). Показывает то же архитектурное решение, но принятое явно и зафиксированное.

## Статус

Proposed (учебный пример для лекции 4, не решение для реального пилота `pe2030-air-quality-pilot`).

## Контекст

Пилотный инкремент сопоставляет жалобы жителей с показаниями двух типов источников: официальных станций (PM2.5, µg/m³, часовое среднее) и недорогих датчиков (VOC-индекс, условная шкала 0–500, мгновенное значение). Эти источники измеряют разные физические величины разными методами и с разной частотой — они не сопоставимы напрямую (см. пример убедительной панели неверных данных в [лекции 1](../../01_engineer_in_ai_era/assets/misleading_dashboard_example.md)).

Чтобы classify() могла сравнить показание с порогом на одной шкале, нужно каким-то образом привести значения к общему виду. Решение о том, как именно это делать и где это решение разместить в коде, архитектурно значимо: от него зависит каждый компонент, использующий сравнение, и оно станет дороже менять, если растянется по всей кодовой базе.

## Рассмотренные альтернативы

**A. Зашить приведение к общей шкале внутри функции сравнения** (так поступил AI в [`ai_generated_pilot_code.md`](ai_generated_pilot_code.md)). Быстро реализовать, но предположение о коэффициенте пересчёта невидимо, не тестируется отдельно и его придётся искать внутри логики сравнения при каждом изменении источников.

**B. Вынести нормализацию в отдельный, явно названный модуль-адаптер на источник данных**, с метаданными (единицы, частота, статус калибровки, точность) и явной пометкой, что показатели разных типов сопоставляются лишь приближённо.

**C. Не приводить величины к общей шкале вообще**, а классифицировать каждый источник по собственному, независимо откалиброванному порогу без попытки сравнения между источниками. Наиболее осторожный вариант, но не отвечает требованию «сопоставить жалобу с ближайшим показанием» в исходной формулировке из лекции 3.

## Решение

Выбран вариант **B**: нормализация выносится в отдельный модуль `source_adapters`, один адаптер на тип источника.

```python
@dataclass
class SourceAdapter:
    source_type: str
    unit: str
    to_reference_scale: Callable[[float], float]
    is_approximate: bool
    calibration_note: str


OFFICIAL_STATION = SourceAdapter(
    source_type="official_station",
    unit="µg/m³",
    to_reference_scale=lambda v: v,
    is_approximate=False,
    calibration_note="прямое измерение PM2.5, эталонная шкала",
)

LOW_COST_SENSOR = SourceAdapter(
    source_type="low_cost_sensor",
    unit="index_0_500",
    to_reference_scale=lambda v: v * 0.12,
    is_approximate=True,
    calibration_note=(
        "грубое приближение к эквиваленту PM2.5; коэффициент не откалиброван "
        "лабораторно, использовать только для приоритизации, не для точных выводов"
    ),
)
```

Функция классификации явно проверяет `is_approximate` и включает это в результат:

```python
def classify(complaint, measurements, adapters):
    nearest, adapter = find_nearest_with_adapter(complaint, measurements, adapters)
    if nearest is None:
        return {"status": "недостаточно данных"}
    value = adapter.to_reference_scale(nearest.value)
    return {
        "status": "отклонение" if value > 35.0 else "обычный уровень",
        "approximate": adapter.is_approximate,
        "calibration_note": adapter.calibration_note,
    }
```

## Обоснование

- Предположение о коэффициенте пересчёта стало видимым — оно находится в одном явно названном месте, а не растворено внутри логики сравнения.
- Результат явно сообщает, была ли классификация основана на приближении — это можно показать специалисту экологической службы без необходимости читать исходный код.
- Добавление нового источника данных требует создать новый `SourceAdapter`, а не искать и модифицировать функцию сравнения.

## Условия пересмотра

- Появится второй недорогой источник данных с иной шкалой или измеряемым веществом.
- Результат нужно будет предъявить внешней аудитории (общественному совету, горсовету) — тогда потребуется явная методология, а не «использовать только для приоритизации».
- Появится возможность лабораторной калибровки датчиков — коэффициент `0.12` должен быть заменён обоснованным значением или удалён в пользу прямого измерения того же вещества.

## Последствия

Реализация варианта B требует немного больше кода и времени, чем вариант A. Команда должна явно решить, оправдана ли эта дополнительная работа для одноразового пилотного скрипта — это открытый вопрос, вынесенный в лекцию для обсуждения, а не готовый ответ.

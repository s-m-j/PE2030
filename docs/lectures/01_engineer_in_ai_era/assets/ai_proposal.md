# Предложение AI: система мониторинга качества воздуха

> **Роль в лекции:** показывается студентам в блоке 1 сценария как единственный артефакт — без промпта и без комментариев. Промпт и разбор ограничений открываются только в блоке 3 и далее, если аудитория сама не задаст вопрос «а как это было сделано».

## Промпт

```
Мы — муниципальный экологический центр. Жители одного района регулярно жалуются:
"По вечерам снова пахнет химией". Показания наших датчиков не демонстрируют
превышения нормативов. Руководитель просит быстро сделать систему, которая
будет собирать данные, показывать их на карте, автоматически обнаруживать
превышения, анализировать тенденции и готовить отчёты.

Спроектируй архитектуру, схему данных, набросай API для детекции превышений
и опиши интерфейс дашборда.
```

## Ответ AI (как показывается студентам)

### Архитектура

```text
┌─────────────────┐     ┌──────────────────┐     ┌───────────────────┐
│  Data Ingestion  │────▶│  Time-Series DB   │────▶│   Alerting Service │
│  (station APIs,  │     │  (PostgreSQL +     │     │  (threshold check, │
│   sensor feeds)  │     │   TimescaleDB)     │     │   notifications)   │
└─────────────────┘     └──────────────────┘     └───────────────────┘
                                  │                          │
                                  ▼                          ▼
                         ┌──────────────────┐     ┌───────────────────┐
                         │   Report Service   │     │   Web Dashboard    │
                         │  (weekly/monthly)  │     │  (map + charts)    │
                         └──────────────────┘     └───────────────────┘
```

- Ingestion-сервис опрашивает станции и датчики по расписанию (каждые 5–15 минут) и пишет нормализованные измерения в единую таблицу.
- Alerting-сервис сравнивает свежие значения с пороговыми нормативами и рассылает уведомления при превышении.
- Dashboard — веб-приложение с картой станций, графиками временных рядов и лентой оповещений.

### Схема данных

```sql
CREATE TABLE measurements (
    id            BIGSERIAL PRIMARY KEY,
    station_id    TEXT NOT NULL,
    pollutant     TEXT NOT NULL,        -- 'PM2.5', 'PM10', 'NO2', 'VOC_INDEX', ...
    value         DOUBLE PRECISION NOT NULL,
    unit          TEXT NOT NULL,        -- 'µg/m³', 'ppb', 'index_0_500', ...
    measured_at   TIMESTAMPTZ NOT NULL,
    source_type   TEXT NOT NULL         -- 'official_station', 'low_cost_sensor', 'model'
);

CREATE INDEX idx_measurements_station_time
    ON measurements (station_id, measured_at DESC);
```

### Фрагмент кода: детекция превышения

```python
THRESHOLDS = {
    "PM2.5": 35.0,   # µg/m³, суточная норма
    "PM10": 50.0,    # µg/m³
    "NO2": 40.0,     # µg/m³
}

def check_exceedance(measurement: Measurement) -> Alert | None:
    limit = THRESHOLDS.get(measurement.pollutant)
    if limit is None:
        return None
    if measurement.value > limit:
        return Alert(
            station_id=measurement.station_id,
            pollutant=measurement.pollutant,
            value=measurement.value,
            limit=limit,
            severity="high" if measurement.value > limit * 1.5 else "medium",
        )
    return None
```

### Интерфейс дашборда (описание для макета)

- **Карта города** с маркерами станций и датчиков; цвет маркера — по индексу качества воздуха (зелёный/жёлтый/оранжевый/красный).
- **Панель графиков** справа: временной ряд по выбранной станции, с наложением нескольких загрязняющих веществ на одной оси времени.
- **Лента оповещений**: список последних превышений с меткой времени и станцией.
- **Раздел отчётов**: экспорт недельной/месячной сводки в PDF.

## Что скрыто за этим ответом (не показывать в блоке 1 — материал для блока 3 и далее)

Ответ технически связен, использует общепринятые практики (time-series БД, отдельный alerting-сервис, понятная схема) и производит впечатление профессионально сделанной системы. При этом он молча предполагает то, что в реальности неизвестно:

- **Пороги те же, что и в существующих датчиках.** Именно эти пороги, по условию кейса, уже не показывают превышения — то есть система с высокой вероятностью повторит ту же слепую зону, только быстрее и с более убедительным интерфейсом.
- **Разные типы источников считаются сопоставимыми.** `source_type` в схеме различает `official_station`, `low_cost_sensor`, `model`, но ни архитектура, ни код не показывают, как система учитывает разную точность, частоту и единицы измерения этих источников при сравнении или наложении на одном графике (см. [`misleading_dashboard_example.md`](misleading_dashboard_example.md) — это ровно та ошибка, которую иллюстрирует второй артефакт).
- **Отсутствие превышения приравнено к отсутствию проблемы.** Модель угроз не различает «датчик не поймал событие» и «события не было».
- **Не указано, кто и на основании чего принимает решение о расследовании жалобы** — alerting-сервис уведомляет, но не определено, что происходит с уведомлением дальше.
- **Не сформулированы цена ложной тревоги и цена пропущенного события** — пороги и уровни severity (`"high"`/`"medium"`) заданы произвольно, без обоснования.
- **Никак не завязано на исходную жалобу жителей** — система не помогает сопоставить время и место жалобы с показаниями ближайших источников.

Ни один из этих пропусков не виден в самой архитектуре, схеме или коде — все они технически корректны. Пропуски видны только при сопоставлении артефакта с исходной проблемой.

```python
_CACHE: dict[str, tuple[float, Any]] = {}

def cached(ttl_seconds: int = 300):
    def decorator(fn):
        def wrapper(*args, **kwargs):
            key = f"{fn.__name__}:{args}:{kwargs}"
            if key in _CACHE:
                cached_at, value = _CACHE[key]
                if time.time() - cached_at < ttl_seconds:
                    return value
            result = fn(*args, **kwargs)
            _CACHE[key] = (time.time(), result)
            return result
        return wrapper
    return decorator
```

Применено к `find_nearest_with_adapter` в `complaints.py`: `@cached(ttl_seconds=300)`.

## Что скрыто за этим PR (не показывать до блока 2 — материал для разбора)

- **PR смешивает минимум четыре независимых изменения** в одном: реализацию ADR (фрагменты 1–2), рефакторинг стиля (фрагмент 3), новую функциональность кеширования, о которой никто не просил (фрагмент 4), и переименования по всему кодовому дереву (фрагмент 5).
- **Кеширование меняет поведение системы**, а не только его оформление: `find_nearest_with_adapter` теперь может вернуть результат пятиминутной давности, если данные за это время обновились — ни ADR, ни исходный промпт не обсуждали, допустимо ли это для инкремента, где корректность сопоставления жалобы с показанием является критерием приёмки из лекции 3.
- **Описание PR не разделяет эти четыре части** ни по важности, ни по срочности — оно перечисляет их одним абзацем, как будто это одно связанное изменение.
- **Ни один коммит не ссылается на ADR явно** — связь с решением из лекции 4 нужно устанавливать самостоятельно, читая код, а не из истории изменений.
- **68 из 96 удалённых и 412 добавленных строк относятся не к реализации ADR**, а к сопутствующему рефакторингу, кешированию и переименованиям — то есть содержательное ядро изменения (адаптеры и логика сопоставления) составляет меньшую часть диффа, которую предстоит найти.

# Контрпример: то же решение, реализованное за четыре шага

> **Роль в лекции:** контрпример к [`ai_generated_giant_pr.md`](ai_generated_giant_pr.md), используется в блоке 7 сценария ([../script.md](../script.md)). Показывает тот же результат, но декомпозированный на независимо проверяемые изменения.

Тот же промпт был переформулирован: «Реализуй ADR о нормализации данных отдельными, независимо проверяемыми шагами. Каждый шаг — отдельное изменение со своим сообщением, ссылающимся на ADR. Не предлагай изменений за пределами ADR — если видишь возможность для улучшения, опиши её отдельно, не включая в реализацию».

## PR 1 — `source_adapters.py`: адаптеры источников данных

**Сообщение:** `Add source adapters per ADR: data normalization (adr_data_normalization.md)`

Содержит только `SourceAdapter`, `OFFICIAL_STATION`, `LOW_COST_SENSOR` и тесты на них (6 тестов). Не затрагивает существующий код сопоставления.

**Ревью занимает 5 минут:** изменение ограничено одним новым файлом, прямо реализующим то, что описано в ADR.

## PR 2 — `complaints.py`: использование адаптеров в сопоставлении

**Сообщение:** `Use source adapters in classify() per ADR (adr_data_normalization.md); depends on PR#1`

Заменяет прежнюю функцию `to_comparable_scale` на вызов `adapter.to_reference_scale`, добавляет поле `approximate` в результат классификации. 4 теста, включая тест именно на ветку с недорогим датчиком (то, что не было покрыто в исходном коде из лекции 4).

**Ревью занимает 10 минут:** изменение видно целиком, явно связано с PR#1 и с ADR, покрывает ранее непроверенный случай.

## PR 3 (отдельное предложение, не смешано с реализацией ADR) — кеширование `find_nearest_with_adapter`

**Сообщение:** `Proposal: cache find_nearest_with_adapter lookups (NOT required by ADR — needs discussion)`

**Описание:** при профилировании сопоставления на полном месяце данных функция поиска ближайшего измерения оказалась заметным узким местом. Предлагается кеширование с TTL 5 минут. **Открытый вопрос для команды:** приемлемо ли для критерия приёмки из лекции 3 (сопоставление жалобы с показанием), что результат может быть до 5 минут устаревшим? Требует явного решения, прежде чем сливать с основной веткой.

**Ревью:** может быть отложено или отклонено отдельно от PR#1–2 — функциональность инкремента не зависит от него.

## PR 4 (отдельное предложение, не смешано с реализацией ADR) — форматирование и переименования

**Сообщение:** `Style: reformat data_io.py, rename make_report→build_report, fmt_row→format_row (cosmetic, no behavior change)`

**Описание:** приведение стиля модуля данных и отчётов к текущим соглашениям проекта. Не влияет на поведение. Может быть смёржено в любое время независимо от остальных изменений или отклонено без последствий для инкремента.

---

## Чем это отличается от единого PR

| | `ai_generated_giant_pr.md` | Декомпозированный вариант |
|---|---|---|
| Число изменений | 1 | 4, с явной зависимостью PR#2 от PR#1 |
| Связь с ADR | нужно устанавливать самостоятельно | указана в каждом сообщении |
| Посторонний scope | смешан с реализацией ADR | вынесен в отдельные PR#3–4 с явной пометкой |
| Время на содержательное ревью PR, реализующего ADR | практически невозможно уложиться в 20 минут | около 15 минут суммарно на PR#1–2 |
| Поведенческий риск (кеширование) | скрыт внутри общего изменения | явно вынесен как открытый вопрос, требующий решения команды |

Суммарное время команды на весь набор изменений не обязательно меньше, чем на один большой PR, — но оно распределено предсказуемо и не мешает вовремя понять и принять минимально необходимую часть (PR#1–2) до встречи с руководителем центра, оставив PR#3–4 на потом.
